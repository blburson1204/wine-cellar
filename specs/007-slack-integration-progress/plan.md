## Summary

Build a Slack notification system for SpecKit workflows with dual-mode
architecture: webhook mode for automatic personal notifications via Claude Code
hooks, and MCP server mode for team-facing tools. New package at
`packages/slack-mcp/` following jira-mcp patterns. Modify existing SpecKit hooks
to fire notifications on phase transitions, task completions, and milestones.

**Language/Version**: TypeScript 5.8+, ES2022 target, Node16 module resolution
**Primary Dependencies**: @modelcontextprotocol/sdk ^1.12.1, zod ^3.24
**Storage**: N/A (stateless — no database, no sync state file) **Testing**:
Vitest 3.x, 80% coverage thresholds **Target Platform**: Node.js 18+ (Claude
Code runtime) **Project Type**: Monorepo package (`packages/slack-mcp/`)
**Constraints**: Notifications must never block SpecKit pipeline (<5s timeout)
**Scale/Scope**: Solo developer, ~3-5 notifications per SpecKit session

## Phase 1: Design & Contracts

**Status**: COMPLETE

**Artifacts generated**:

- [data-model.md](data-model.md) — Configuration model, event types,
  SpecProgress read model
- [contracts/slack-mcp-tools.ts](contracts/slack-mcp-tools.ts) — MCP tool
  input/output interfaces, notification event types, config types
- [quickstart.md](quickstart.md) — Setup guide for both modes

**Key design decisions**:

1. **No persistent state** — Unlike jira-mcp, no sync file. Each notification is
   independent.
2. **Dual entry points** — `index.ts` for MCP server mode, `notify.ts` for
   hook-triggered webhook mode.
3. **Shared internals** — Both modes use the same config, formatter, and
   slack-client modules.
4. **Extensible tool registration** — Tool handlers are modular functions, easy
   to add `update_task_status` later.
